# AI 기술 글 편집 가이드

이 문서는 AI 발표 글을 빠르게 늘리는 것보다 독자가 판단하고 재현할 수 있는 콘텐츠를 만드는 기준을 정의한다.

## 1. 글을 쓰기 전에 결정할 것

모든 글은 다음 질문에 한 문장으로 답할 수 있어야 한다.

- **독자**: 이 글을 읽을 구체적인 개발자는 누구인가?
- **과제**: 독자가 글을 읽은 뒤 무엇을 이해하거나 결정할 수 있어야 하는가?
- **형식**: 브리핑인가, 심층 분석인가, 튜토리얼인가, 사용법인가?
- **근거**: 공식 발표, 논문, 코드, 직접 실험 가운데 무엇을 확인했는가?
- **범위**: 무엇을 직접 검증하지 않았는가?

튜토리얼, 사용법, 참조, 설명은 서로 다른 독자 요구를 다룬다는 [Diátaxis](https://diataxis.fr/)의 구분을 따른다. AI 발표 글은 기본적으로 설명에 속한다. 실행 가능한 결과와 검증 환경이 없으면 튜토리얼이라고 부르지 않는다.

## 2. 두 가지 발행 형식

### 브리핑

발표 당시의 핵심을 3~5분 안에 확인하려는 독자를 위한 글이다.

- 한 문장 요약
- 발표 범위와 제공 상태
- 기술적으로 달라진 지점
- 중요성 및 한계
- 공식 1차 출처

모든 글에 같은 일반론이나 체크리스트를 붙여 분량을 늘리지 않는다. 해당 발표에 고유한 정보만 남긴다.

### 심층 분석

기술 또는 제품 결정을 내려야 하는 독자를 위한 글이다.

- 글쓴이의 핵심 판단과 독자의 질문
- 기술 구성 요소와 작동 흐름
- 발표가 바꾼 것과 바꾸지 않은 것
- 주제별 설계 교훈
- 직접 재현할 수 있는 평가 순서
- 글쓴이의 판단과 검증 범위
- 관련 글과 공식 1차 출처

심층 분석은 단순 요약보다 새로운 비교, 구조화, 적용 판단을 제공해야 한다. 이는 독자 우선 콘텐츠, 독창적 분석, 명확한 출처와 작성 배경을 권장하는 [Google Search의 people-first 콘텐츠 가이드](https://developers.google.com/search/docs/fundamentals/creating-helpful-content)에 따른다.

## 3. 근거 수준을 문장에 표시하는 법

다음 표현을 구분한다.

- **공식 발표**: “회사는 …라고 발표했다”, “공식 평가에서는 …를 보고했다.”
- **공개 자료에서 확인**: “공개된 구조는 …로 구성된다.”
- **직접 검증**: 실행 환경, 데이터, 명령, 결과를 함께 제시할 때만 “확인했다”라고 쓴다.
- **편집 판단**: “내 판단은 …다”라고 주체를 드러내고 사실과 분리한다.

직접 실행하지 않은 벤치마크를 재현했다고 쓰지 않는다. AI가 자료 구조화나 초안 작성에 사용됐으면 독자가 궁금해할 위치에 제작 방식을 밝힌다. 이는 [Google의 생성형 AI 콘텐츠 가이드](https://developers.google.com/search/docs/fundamentals/using-gen-ai-content)와 Who·How·Why 원칙을 반영한다.

## 4. 문장과 구조

- 한 문장에는 핵심 생각 하나만 둔다.
- 행위 주체를 숨기지 않고 능동형을 우선한다.
- 제목은 내용을 정확히 설명하고 과장하거나 빈 호기심을 유도하지 않는다.
- 소제목만 읽어도 글의 논리와 독자의 다음 행동이 보이게 한다.
- 순서가 중요하면 번호 목록, 순서가 없으면 글머리표를 사용한다.
- 표와 목록 앞에는 무엇을 비교하는지 설명하는 문장을 둔다.
- 긴 글은 목차와 관련 글 링크로 탐색 경로를 제공한다.

세부 문장 기준은 Google Technical Writing의 [짧은 문장](https://developers.google.com/tech-writing/one/short-sentences), [능동형](https://developers.google.com/tech-writing/one/active-voice), [목록과 표](https://developers.google.com/tech-writing/one/lists-and-tables) 원칙을 따른다. 제목은 [Google Search 제목 가이드](https://developers.google.com/search/docs/appearance/title-link)처럼 페이지별로 고유하고 구체적으로 작성한다.

## 5. 코드와 재현 절차

코드 예제는 독자가 복사할 수 있다고 가정한다.

- 실행에 필요한 런타임, 버전, 의존성, 환경 변수를 밝힌다.
- 비밀 값은 예시 환경 변수로 표시하고 실제 키를 넣지 않는다.
- 실행할 수 없는 의사 코드는 명시적으로 의사 코드라고 표시한다.
- 핵심 개념과 무관한 프레임워크나 의존성을 제거한다.
- 성공 출력뿐 아니라 예상 실패와 확인 방법을 제공한다.
- 보안상 위험하거나 폐기된 API를 짧다는 이유로 사용하지 않는다.

이 기준은 복사된 코드가 실제 서비스에 들어갈 수 있음을 전제로 하는 [MDN 코드 예제 가이드](https://developer.mozilla.org/en-US/docs/MDN/Writing_guidelines/Code_style_guide)를 따른다.

## 6. 발행 전 체크

1. 제목과 설명이 실제 본문 약속과 일치하는지 확인한다.
2. 날짜, 제품 제공 상태, 수치의 조건을 공식 1차 출처에서 확인한다.
3. 발표사의 주장과 편집 판단, 직접 검증을 분리한다.
4. 반복 문구와 다른 글에도 그대로 붙일 수 있는 문단을 삭제한다.
5. 관련 글 2~3편을 설명적인 링크 문구로 연결한다.
6. 링크, MDX 렌더링, 모바일 가독성을 확인한다.
7. 콘텐츠 검증, 테스트, 린트, 프로덕션 빌드를 실행한다.

## 7. 유지보수

- 과거 발표일은 그대로 두고 실질적인 수정이 있으면 `updated` 날짜를 기록한다.
- 제공 상태나 사양이 바뀌면 과거 문장을 현재 사실처럼 고치지 말고 발표 당시 상태와 현재 상태를 구분한다.
- 링크가 사라지면 가능한 한 같은 기관의 공식 문서 또는 저장소로 교체한다.
- 대표 심층 분석은 분기마다 출처, 내부 링크, 직접 검증 가능성을 다시 점검한다.
