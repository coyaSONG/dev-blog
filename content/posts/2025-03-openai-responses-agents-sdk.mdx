> **핵심 판단**: Responses API와 Agents SDK의 의미는 “에이전트”라는 이름보다 도구 호출, 상태, 추적을 애플리케이션의 1급 구성 요소로 만든 데 있다.

챗 완성 API 위에 도구 루프를 직접 만들면 호출 ID, 재시도, 승인, 컨텍스트 압축, 로그 연결을 팀마다 다시 구현하게 된다. 2025년 발표는 웹 검색·파일 검색·컴퓨터 사용 같은 도구와 오케스트레이션 SDK를 하나의 개발 표면으로 묶었다. 모델의 답변 문자열보다 실행 과정 전체를 다루는 API로 무게중심이 옮겨간 것이다.

## 발표를 한눈에 보기

- **발표 주체**: OpenAI
- **공식 발표일**: 2025-03-11
- **대상**: Responses API and Agents SDK
- **발표 당시 상태**: Responses API, 내장 도구와 Agents SDK 공개
- **이 글의 질문**: 모델 호출을 상태 있는 작업 실행으로 바꿀 때 어떤 경계를 애플리케이션이 소유해야 하는가?

다중 에이전트 handoff, tracing, guardrail을 제공하며 Assistants API의 장기 후속 경로를 제시했다.

## 기술 구조: 무엇이 실제로 달라졌나

모델 응답과 도구 호출 이벤트를 하나의 상태 있는 인터페이스로 다루고 실행 추적을 SDK의 일급 기능으로 만든다.

Responses API는 텍스트 생성과 도구 사용을 하나의 응답 흐름으로 표현하고, Agents SDK는 에이전트 간 위임, 가드레일, 추적을 구성하는 계층을 제공했다. 내장 도구는 검색이나 컴퓨터 조작 같은 반복 구현을 줄인다.

그러나 SDK가 비즈니스 트랜잭션을 대신 보장하지는 않는다. 모델이 같은 도구를 재호출할 수 있고 네트워크 타임아웃 뒤 실제 작업 성공 여부가 불명확할 수 있다. 외부 효과가 있는 호출에는 멱등성 키, 상태 머신, 보상 동작, 사람 승인이 여전히 필요하다.

> **작동 흐름**: 사용자 목표 수신 → 모델이 응답 또는 도구 호출 결정 → 정책 검사 → 도구 실행과 결과 연결 → 필요 시 다른 에이전트에 위임 → 추적과 최종 응답 저장

이 구조에서 개발자가 가져갈 설계 원칙은 다음과 같다:

- 도구 실행을 단순 함수 호출이 아니라 재시도와 중복 실행이 가능한 분산 트랜잭션으로 취급한다.
- 모델 추적 로그와 애플리케이션 감사 로그를 공통 작업 ID로 연결하되 민감한 원문은 최소 저장한다.
- 에이전트 수를 역할 이름으로 늘리기 전에 권한, 평가 기준, 컨텍스트 경계가 실제로 다른지 확인한다.

## 발표가 바꾼 것과 바꾸지 않은 것

에이전트 개발이 프롬프트 연결에서 관측 가능하고 운영 가능한 소프트웨어 프레임워크로 이동했다.

이 발표는 에이전트 개발의 경쟁 지점을 프롬프트에서 런타임으로 옮겼다. 팀은 도구 스키마와 핸드오프, 트레이싱, 평가를 코드로 관리할 수 있게 됐다. 반면 특정 공급자의 내장 도구와 상태 형식에 의존하는 잠금 효과도 아키텍처 검토 대상이 됐다.

다만 다음 문제까지 해결됐다고 확대 해석해서는 안 된다:

- SDK가 제공하는 추적은 업무 데이터의 정확성이나 도구 결과의 정당성을 자동 검증하지 않는다.
- 여러 에이전트로 나누면 품질이 자동 상승하지 않으며 지연·비용·실패 지점은 늘 수 있다.
- 컴퓨터 사용 도구는 UI 변경과 프롬프트 인젝션에 취약하므로 고위험 작업을 무감독으로 실행하면 안 된다.

## 직접 평가한다면 이렇게 본다

발표사의 종합 점수 하나를 재현하는 것보다, 실제 제품의 입력과 실패 비용에 맞춘 평가가 더 유용하다. 이 주제라면 다음 순서로 확인한다:

- 성공 경로뿐 아니라 도구 타임아웃, 중복 결과, 부분 성공, 권한 거부를 주입한다.
- 단일 모델 루프와 다중 에이전트 구성을 같은 작업 성공률·비용·완료 시간으로 비교한다.
- 각 도구 호출이 사용자 의도와 허용 범위 안에 있었는지 샘플 기반으로 감사한다.
- 모델·도구 버전이 바뀔 때 고정된 작업 세트로 회귀 평가를 자동 실행한다.

에이전트 평가는 최종 답변의 문장 품질보다 외부 상태가 올바르게 바뀌었는지를 기준으로 해야 한다. “완료했다”는 모델 메시지를 성공 신호로 사용하면 안 된다.

## 내 판단

이 도구군은 에이전트 프로토타입을 빠르게 만들지만, 프로덕션의 본질은 여전히 상태와 권한이다. SDK는 오케스트레이션 보일러플레이트를 줄여줄 뿐, 트랜잭션 경계를 대신 결정하지 않는다. 중요한 작업일수록 도구 호출을 애플리케이션의 명시적 상태 머신 안에 가두는 설계가 필요하다.

내장 도구가 권한·데이터 거버넌스를 자동 해결하지 않으며 API 세대 전환에 따른 마이그레이션 계획이 필요했다.
